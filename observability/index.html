Post · May 2026

I spent 1 day building out in-house observability, and now I can diagnose and fix bugs 100x faster.

Over the past 3 years, I've tried out a few different monitoring tools, ultimately landing on Sentry for error monitoring and New Relic for application performance monitoring. They served their purpose providing high level reporting, but I wanted observability into everything happening across our platform.

After a couple of earnest tries to get these tools to fit my use case, I gave up. That was until a couple months ago, when I decided to build in-house the tool I wanted. Using one coding agent, I ran an audit on every call, message, and cron job that is handled across our platform. I leveraged another agent to implement an observability table and trigger a new log for each of these actions. Over the next two weeks, our platform gathered and logged millions of events.

To make sure our database didn't balloon, I pushed another release to archive stale observability data to S3 and prune our Postgres database. We were able to see the fruits of this rollout immediately. An agent went through our logs to figure out a double-trigger click issue some of our users were experiencing. Within minutes, it had spanned our logs, pinpointed the trail showcasing what kept happening, and suggested a VPN-related idiosyncrasy as the root cause. Later that day, I shipped a fix with idempotency keys. High-fidelity observability leveraged by coding agents has completely changed our bug-fixing workflow. Now, a bug can be fixed in minutes, not hours.

The next intuitive progression of this bug fixing workflow is self-healing coding agents: creating a recurring cron job that goes through logs each day to proactively find, diagnose, and present solutions to unexpected behavior.

This engineering workflow is a force multiplier that I highly encourage all engineering teams to invest in. I've shared an agent prompt that should be able to get you 90%+ of the way there — you may need to tweak it given your own stack considerations. Reach out if you have any questions or thoughts on this implementation.

The Prompts

Build a Custom Observability System for Your App

Three prompts to hand a coding agent (Claude Code, Cursor, etc.) in order: an audit pass to find every touchpoint, an implementation pass that builds the whole system, and an optional retention/pruning pass. Reference stack is Next.js + Prisma + PostgreSQL with a sibling Node service — translate paths and framework idioms to your own.

Prompt 1 — Audit (recon only; do not modify code)

Audit my codebase for every place a custom observability system would need
to capture an event. Output a report I can hand to another agent that will
implement the system. Do NOT make any code changes — this is a report-only
pass.

For each category below, search the codebase and list file:line locations.
For each finding, include: file path + line, category, what kind of event
would be emitted (type, key fields), and any notes on edge cases or
intentional skips.

Categories:

1. **API auth middleware(s)** — Find the main auth/withAuth wrapper that
   every authenticated API route flows through. List its location. Also
   look for any custom auth wrappers, public-API wrappers, webhook
   receivers, or handler patterns that BYPASS the main middleware — these
   need to be wrapped separately.

2. **Cron jobs / scheduled tasks** — Find every scheduled task (cron
   schedule patterns, setInterval, scheduler handlers, GitHub Actions
   workflows, AWS EventBridge rules, Aptible scheduled containers). For
   each: taskName, what it does, and whether it currently has any logging.

3. **Outbound messages** — Find every place the app sends an email, SMS,
   push notification, Slack message, or any user-facing communication.
   Note the message subtype, recipient type, and sender function.

4. **Cross-service HTTP calls** — Find every fetch/axios/got call to
   external APIs or sibling services. Distinguish internal sibling calls
   (success logged on receiving side) from true external API calls
   (success and failure logged here).

5. **Integration touchpoints** — Find every place the app reads or writes
   to an external integration (EHR, billing, referral, third-party API).
   These need a separate IntegrationEvent record with direction, status,
   and full request/response payloads.

6. **Idempotency middleware** — If idempotency middleware exists, find
   the cached-response replay paths. These need IDEMPOTENCY_REPLAY events.

7. **Background workers / queue consumers** — Find every queue worker,
   job processor, or background task. Note their entry points.

8. **Websocket / real-time handlers** — If a websocket server or SSE
   endpoints exist, find every message handler routed by message type.

9. **Existing observability** — Note any existing logging
   (console.log, Sentry, New Relic custom events, structured logging).
   Flag anything sensitive being logged un-sanitized (password, token,
   secret, ssn, creditCard, apiKey).

For each finding, output a row:
  - file:line
  - Category (1-9)
  - Recommended event shape (type, key fields)
  - Notes / edge cases / intentional skip reason

Finish with a one-paragraph summary: roughly how many events/day this
system will generate, which areas are highest-value, and any architectural
gotchas (e.g. "the public-API wrapper doesn't share auth middleware with
the main app — needs separate instrumentation").

Prompt 2 — Implementation (build the whole system from the audit findings)

Implement a custom observability system based on the audit report from
Prompt 1. This is a single comprehensive build. Below is the full
specification — work through it top to bottom. Reference the audit findings
to know which routes/crons/messages/integrations need wiring.

## Part A — Database schema (Prisma)

Add two new Prisma models.

Model 1: SystemEvent — for API calls, cron jobs, automated messages,
idempotency replays.
Fields:
- id (auto-increment int, primary key)
- type (enum SystemEventType: CRON, MESSAGE, SERVICE_CALL, IDEMPOTENCY_REPLAY)
- result (enum SystemEventResult: SUCCESS, FAILURE)
- service (string, nullable) — which service emitted it
- environment (string, nullable) — "local" | "dev" | "production"
- route (string, nullable)
- method (string, nullable)
- userEmail (string, nullable)
- errorMessage (string, nullable)
- stackTrace (Text, nullable)
- taskName (string, nullable) — for cron jobs
- operation (string, nullable) — sub-operation within a task
- durationMs (int, nullable)
- itemsProcessed (int, nullable)
- metadata (Json, nullable) — free-form structured context
- createdAt (DateTime, default now())
- updatedAt (DateTime, @updatedAt)
Indexes: [type], [service], [createdAt]

Model 2: IntegrationEvent — for inbound/outbound integration traffic.
Fields:
- id (auto-increment int, primary key)
- type (enum IntegrationEventType: REFERRAL, REPORTING, BILLING) — adjust
  enum values to match the audit findings
- direction (enum IntegrationDirection: INBOUND, OUTBOUND)
- status (enum IntegrationStatus: RECEIVED, ACKED, SENT, DELIVERED, FAILED, REJECTED)
- endpoint (string, nullable)
- statusCode (int, nullable)
- requestPayload (Json, nullable)
- responsePayload (Json, nullable)
- workspaceId (int, nullable, relation to tenant model — replace with whatever yours is)
- createdAt, updatedAt, deletedAt (DateTime; deletedAt nullable for soft delete)
Indexes: [workspaceId], [type, direction, status, createdAt] (composite), [deletedAt]

Edit schema.prisma and generate the migration. Print the migration command
for me to run — do NOT auto-run `prisma migrate dev`.

## Part B — Capture utility

Create src/server/system-events.ts (adjust path to wherever your server-only
code lives).

Export async `logSystemEvent(params)` that writes one SystemEvent row.

Critical requirements:
1. Wrap the entire body in try/catch. NEVER throw — observability failures
   must not break parent requests. Console-log on failure and swallow.
2. Default `service` to the main app name if not provided.
3. Resolve `environment` automatically from a helper that reads env vars.
4. If `error` is passed:
   - Error instance → extract .message and .stack
   - Otherwise → String() it as the message, no stack
5. Pass `metadata` straight through as Prisma Json.

Also export:
- `sanitizeBody(body)` — shallow-copy and replace fields whose name
  (case-insensitive) contains "password", "token", "secret", "apiKey"
  with "[REDACTED]". Returns undefined if body is null or not an object.
- `captureResponseBody(res)` — patches res.json and res.send to capture
  the outgoing body. Returns a getter `() => unknown`. We need this
  because handlers can return 5xx without throwing.
- `errorFromResponseBody(statusCode, body)` — if statusCode >= 500, build
  an Error from body.error / body.message / body.details / JSON.stringify(body).
  Returns undefined for non-5xx.
- `extractResourceIds(url)` — parse "/api/foo/patients/123/sessions/456"
  into { patientsId: "123", sessionsId: "456" }. Detect numeric segments
  following non-numeric segments.

Type the params with a TypeScript interface matching SystemEvent fields
(excluding id, environment, createdAt, updatedAt, and the resolved
errorMessage/stackTrace which come from `error`).

## Part C — Middleware integration (auto-coverage for API routes)

Wire `logSystemEvent` into the main auth middleware identified in the
audit. For each authenticated handler invocation:

1. startTime = Date.now() before calling the handler
2. Wrap res with captureResponseBody
3. Call the handler inside try/catch
4. On success path:
   - Use errorFromResponseBody(res.statusCode, getResponseBody()) to detect
     handler-returned 5xx (no exception thrown)
   - Call logSystemEvent with type "SERVICE_CALL", result "SUCCESS" if no
     serverError else "FAILURE", route, method, userEmail (from JWT),
     durationMs, error, metadata: { source: "web", userId, resourceIds:
     extractResourceIds(req.url), requestPayload: sanitizeBody(req.body),
     responseStatus: res.statusCode }
5. On caught exception:
   - logSystemEvent with type "SERVICE_CALL", result "FAILURE", same fields,
     error: handlerError
   - If !res.writableEnded, send a 500 with { error: "Internal server error" }

Use `void logSystemEvent(...)` (no await) so observability never blocks the
response. Do NOT wrap the logSystemEvent calls in try/catch — the function
handles its own errors.

If the audit found additional auth wrappers (server-to-server, public-API,
webhook receivers): apply the same pattern. For server-to-server set
metadata.source to "server"; for public-API set it to "public".

If the audit found idempotency middleware: log every replay path with type
"IDEMPOTENCY_REPLAY", result "SUCCESS", metadata.idempotencyKey and
metadata.cachedStatusCode.

## Part D — Cross-service emitter (skip if single-service)

If the audit identified sibling services (websocket server, workers,
schedulers), add a buffered emitter to each.

Create `trackEvent(eventName, result, metadata)` in
src/utils/server-utils.ts (or your equivalent). Behavior:
1. Buffer events in an in-memory array.
2. Flush triggers: buffer reaches 20 events → flush immediately; otherwise
   5-second debounced timer.
3. Flush = POST { events: [...] } to
   ${BASE_URL}/api/v2/server/shared/system-events/ingest with
   Authorization: Bearer ${serverToken}.
4. On flush failure: re-queue up to 50 events (cap to prevent memory
   blowup during prolonged backend outage).

Map well-known eventName strings to SystemEvent types (rename "Zeph*" to
fit your service):
  ZephCronJob     → "CRON"
  ZephMessage     → "MESSAGE"
  ZephServiceCall → "SERVICE_CALL"
  ZephApiCall     → "SERVICE_CALL"
  ZephWebSocket   → undefined (internal, leave null)
  ZephError       → undefined (internal, leave null)

Buffered event shape mirrors the logSystemEvent params interface. Resolve
`environment` the same way as the main backend. Hardcode `service` to this
service's name. Export `flushEvents()` for graceful-shutdown flush.

Then add `trackEvent` calls at every location the audit identified:
- Cron jobs: success at end with { taskName, durationMs, itemsProcessed,
  failureCount }; failure on caught exceptions with same fields plus
  { error, step }
- Outbound messages: success/failure with { messageType, functionName,
  recipientType, ...domain ids }
- Outbound HTTP calls to other services: failures only (success logged on
  receiving side); include { url, method, httpStatus, resourceId,
  requestPayload, taskName }
- Websocket message handlers: success at end with { messageType,
  ...domain ids }; failure on caught exception

Also build the receiving endpoint in the main backend:
POST /api/v2/server/shared/system-events/ingest
- Reject non-POST with 405
- Body: { events: BufferedEvent[] }; reject empty array with 400
- Cap batch at 100 events (slice)
- Reconstruct Error-like object per event from { errorMessage, stackTrace }
- Promise.all + logSystemEvent for each. Default service to sender's name.
- Return { accepted: batch.length }
- Protect with server-to-server auth (NOT user auth)

## Part E — Integration events

For every integration touchpoint the audit identified, wrap the call site
to write an IntegrationEvent with type, direction, status, endpoint,
statusCode, full requestPayload and responsePayload, workspaceId. Both
inbound (we receive) and outbound (we send) directions.

## Part F — Query API

Add three GET endpoints under /api/.../observability/. All require
admin/audit-log permission — gate behind whatever your auth model uses.

1. GET /events — paginated list with filters
   Query params: type, result, environment, service, taskName, userEmail,
   method, source (read from metadata.source), search (free-text against
   route + errorMessage), dateFrom, dateTo (ISO), page (default 1),
   limit (default 50, cap at 200).
   Return { events: [...], total, page, limit }. Order by createdAt desc.

2. GET /summary — aggregated counts since a given window
   Query param: since (parse "24 hours ago" / "7 days ago" / ISO).
   Return { total, byType: { CRON: n, MESSAGE: n, ... },
   byResult: { SUCCESS: n, FAILURE: n }, failureRate }.
   Use prisma.systemEvent.groupBy.

3. GET /cron-status — latest run per distinct taskName in the window
   Query param: since (default "7 days ago").
   For each distinct taskName: { taskName, lastRun, lastResult,
   lastDurationMs, lastItemsProcessed, lastError, totalRuns, failures }.
   Use DISTINCT ON (Postgres) or windowed subquery, enriched with counts.

Add a /integration-events GET endpoint mirroring (1) but for
IntegrationEvent: type, direction, status, workspaceId, from, to, page,
pageSize. Same pagination contract.

## Part G — Dashboard UI

Build an admin page at /management/system-health (or your equivalent admin
route) showing SystemEvent rows with filters.

Filter UI (sidebar or top bar):
- Time range (last 1h / 24h / 7d / 30d / custom)
- Source (Web, Mobile, Websocket Server, Server) — maps to metadata.source
- Type (API Call, Cron Job, Message) — maps to SystemEvent.type
- Method, Result, User (autocomplete by email), Environment, free-text

Result list:
- 50 per page, paginated
- Each row: timestamp, source badge, type label, route or taskName,
  user email, duration, result badge (neutral colors by default — avoid
  destructive red if your design system avoids it)
- Expandable detail panel: full metadata JSON, error message, stack
  trace (collapsed by default, "Show stack trace" link)
- For SERVICE_CALL: show resource IDs parsed from metadata
- For CRON: show taskName, operation, durationMs, itemsProcessed
- For MESSAGE: show recipient type, message subtype

Build a reusable hook `useObservabilityEvents(filters, page, limit)`
calling GET /events, with debounced filter changes, exposing { events,
total, loading, error, refetch }.

Build a formatters module with:
- getEventTypeLabel(type): SERVICE_CALL → "API Call", CRON → "Cron Job",
  MESSAGE → "Automated Message", IDEMPOTENCY_REPLAY → "Idempotency Replay"
- getEventDetail(row): one-line summary (method+route for API calls,
  taskName for crons, message subtype for messages)
- getResourceLabel(row): metadata.resourceIds → "patient #123, session #456"
- formatDuration(ms): "500ms", "1.2s", "1m 30s"
- getSourceLabel(row): metadata.source → "Web" / "Mobile" / "Websocket
  Server" / "Server"

Add an analogous /management/integrations page using the
IntegrationEvent endpoint with filters: type, direction, status, workspace,
date range. Show a workspace summary row (inbound/outbound/failed counts
per workspace).

## Wrap-up

After implementation, re-run the audit from Prompt 1 and confirm every
finding is now wired (or explicitly skipped with a reason). Flag any
remaining anti-patterns:
- `await logSystemEvent(...)` instead of `void logSystemEvent(...)`
- logSystemEvent calls wrapped in their own try/catch (redundant)
- Sensitive fields in metadata.requestPayload not being sanitized
- logSystemEvent calls with no `metadata` field

Prompt 3 — Retention & pruning (optional; only if you already have a cron runner in production)

Add a retention/archival cron for the SystemEvent table. Skip this entire
prompt if I don't already have a cron runner (Aptible scheduled containers,
Vercel Cron, GitHub Actions schedule, node-cron process, AWS EventBridge,
etc.) — do NOT stand up new cron infrastructure just for this.

Endpoint: DELETE /api/v2/server/cron/system-events/cleanup
Protected by server-to-server auth.

Behavior per invocation:
1. Find SystemEvents with createdAt older than RETENTION_DAYS (default 10,
   read from env). Process oldest first.
2. Group rows by UTC day.
3. For each day: serialize rows as gzipped JSONL, upload to S3 at key
   `system-events/<environment>/<YYYY-MM-DD>/batch-<n>.jsonl.gz`.
4. After successful S3 upload, delete the corresponding rows from the DB
   in the same transaction batch.
5. Batch size: 5000 rows per iteration.
6. Time budget: 30 seconds per HTTP call (the cron re-invokes daily, so
   don't need to drain everything in one shot).
7. Safety cap: 200 batches per invocation.

Idempotency: if S3 upload succeeds but DB delete fails (rare), record the
S3 key and have a recovery path that detects orphaned S3 objects on the
next run and skips re-archiving rows that match.

Return { archived: rowCount, batches: batchCount, timeMs }.

Track the cron itself with logSystemEvent type CRON, taskName
"systemEventCleanup", durationMs, itemsProcessed: archived. Yes — the
cleanup cron logs to the table it's cleaning up. Its own rows get archived
next run.

Wire the daily schedule into the existing cron runner (do NOT add a new
cron framework). If I don't have one, stop and tell me — we'll skip this.
